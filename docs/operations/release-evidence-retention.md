# Release evidence retention and audit packages

This repository generates **structured, checksummed audit packages** under the canonical layout `release-evidence/` (see table below). The assembler is [`scripts/release/build_release_evidence_package.py`](../../scripts/release/build_release_evidence_package.py). It **does not invent** missing JSON; `manifest.json` lists each file as present or absent.

## What is generated

| Canonical file | Typical source | When it exists |
| --- | --- | --- |
| `manifest.json` | Generated by the script | Always in a built package |
| `checksums.txt` | SHA-256 of each present file + `manifest.json` | Always |
| `image-metadata.json` | `build-evidence/image-build-metadata.json` | After **Build and Push Images** |
| `promotion-manifest.json` | `build-evidence/promotion-manifest.json` | After build metadata publish |
| `cosign-signing-evidence.json` | `cosign-evidence/cosign-signing-evidence.json` | When cosign evidence is produced in build |
| `sbom-app.cdx.json` / `sbom-goose.cdx.json` | `security-reports/*.cdx.json` | After SBOM step in build metadata job |
| `security-verdict.json` | `security-reports/security-verdict.json` | **Security Release** signal step |
| `staging-deploy-evidence.json` | Staging contract path (if copied into the search root) | Staging deploy / manual merge |
| `backup-evidence.json` | Backup drill JSON (if present in tree) | Production with `run_migration` and backup resolution |
| `production-deploy-evidence.json` | `deployment-evidence/production-deploy-evidence.json` | **Deploy Production** after manifest write |

GitHub Actions uploads **named artifacts** (not Git LFS): each workflow run stores blobs with a retention countdown.

## How long it is retained

| Artifact (examples) | `retention-days` in workflow | Notes |
| --- | ---: | --- |
| `release-audit-package` (build) | **90** | Canonical build-phase audit directory |
| `security-release-audit-package` | **90** | Security slice; `security-verdict` JSON also kept **180** days in the same workflow |
| `production-release-audit-package` | **90** | Production slice from deploy job |
| `production-deploy-evidence` (full tree) | **90** | Full `deployment-evidence/` from **Deploy Production** |
| Raw SBOM / build metadata in build job | **180** | Older pin; longer retention for raw CycloneDX / manifests |

**GitHub limits:** For **public** repositories, artifact retention can be set per artifact up to **400 days** (GitHub documentation). **Enterprise Cloud** and **organization/repository** policies may cap retention **below** that (for example 30 or 90 days). If your org enforces a maximum, the workflow value is silently clamped to that maximum. This is not visible in YAML; check **Settings → Actions → General → Artifact and log retention** (org or repo).

**Implication for immutability:** Artifacts are **immutable for a given run id** until expiry. They are not a legal hold archive. For years-long traceability, export packages (see below).

## How to download

1. **GitHub UI:** Repository → **Actions** → select the workflow run → **Artifacts** → download the zip (e.g. `release-audit-package`, `security-release-audit-package`, `production-release-audit-package`).
2. **GitHub CLI:**  
   `gh run download <RUN_ID> -n release-audit-package -D ./audit`
3. **API:** Use the [Artifacts API](https://docs.github.com/en/rest/actions/artifacts) with `actions:read` scope.

## How to verify checksums

After extracting a package:

1. Recompute SHA-256 for each file listed in `checksums.txt` (format: `<hex>  <filename>`), e.g. on Linux:  
   `sha256sum -c checksums.txt` (GNU coreutils) — note `checksums.txt` includes `manifest.json` last; verify the file is intact before trusting earlier lines.
2. Or verify manually:  
   `sha256sum manifest.json` and compare the first line for `manifest.json` in `checksums.txt`.

The script records **only** files that were copied; it never pads missing entries with fake content.

## When to export to external storage

- **Regulatory or enterprise policy** requires retention **beyond** the GitHub artifact window or org cap.
- **Disaster recovery:** long-term WORM or object store (S3, GCS, Azure Blob) with versioning.
- **Merge audit slices:** download `release-audit-package` (build) and `security-release-audit-package`, then run:  
  `python3 scripts/release/build_release_evidence_package.py --out release-evidence --phase release --root . --merge-from /path/build_pkg --merge-from /path/security_pkg --strict-phase release`  
  from a directory that also contains all required paths, **or** place merged canonical JSONs in one folder and use `--merge-from` for each.

**Optional external steps (not automated here):** upload the zip to a company bucket; attach the workflow run URL and `GITHUB_SHA` in your CMDB; sign the package with an offline key if policy requires an additional layer beyond GitHub and SBOMs.
