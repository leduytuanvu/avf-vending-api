# Processes and capabilities

This repo ships five main Go binaries. Each is a separate process; together they form a modular monolith style deployment (not microservices).

| Process | Entry | Required for | Hard dependencies | Optional / feature flags |
| ------- | ----- | -------------- | ------------------- | ------------------------- |
| **API** | `cmd/api` | HTTP `/v1/*`, health, version, optional gRPC | Postgres; Redis if your config uses it; JWT secrets per `HTTP_AUTH_MODE` | `METRICS_ENABLED`, optional private ops mirror on `HTTP_OPS_ADDR`, `GRPC_ENABLED`, `API_ARTIFACTS_ENABLED` (+ S3 env), MQTT publish/NATS when wiring flags demand it; `TEMPORAL_*` when `TEMPORAL_ENABLED=true` (dials Temporal at startup; see `internal/app/workfloworch`) |
| **Worker** | `cmd/worker` | Outbox dispatch, payment timeout scan, stuck command scan | Postgres | `NATS_URL` for JetStream publish; optional Temporal scheduling for payment-timeout follow-up via `TEMPORAL_ENABLED=true` + `TEMPORAL_SCHEDULE_PAYMENT_PENDING_TIMEOUT=true`; always exposes `/health/live` + `/health/ready` + `/version` on `WORKER_METRICS_LISTEN` (default `127.0.0.1:9091`); `METRICS_ENABLED=true` also exposes `/metrics`; optional ClickHouse mirror via `ANALYTICS_*` (see `ops/ANALYTICS_CLICKHOUSE.md`) |
| **MQTT ingest** | `cmd/mqtt-ingest` | Device telemetry / shadow / command receipts over MQTT | Postgres + MQTT broker env (`internal/platform/mqtt`) | Always exposes `/health/live` + `/health/ready` + `/version` on `MQTT_INGEST_METRICS_LISTEN` (default `127.0.0.1:9093`); `METRICS_ENABLED=true` also exposes `/metrics` |
| **Reconciler** | `cmd/reconciler` | Commerce list passes; optional PSP probe + refund enqueue | Postgres | `RECONCILER_ACTIONS_ENABLED` (+ probe URL, NATS, applier wiring); optional Temporal scheduling for refund/manual-review follow-up via `TEMPORAL_SCHEDULE_REFUND_ORCHESTRATION=true` / `TEMPORAL_SCHEDULE_MANUAL_REVIEW_ESCALATION=true`; always exposes `/health/live` + `/health/ready` + `/version` on `RECONCILER_METRICS_LISTEN` (default `127.0.0.1:9092`); `METRICS_ENABLED=true` also exposes `/metrics` |
| **Temporal worker** | `cmd/temporal-worker` | Executes Temporal workflows/activities for long-running compensation and review follow-up | Postgres; Temporal frontend; NATS refund-review sink | Always exposes `/health/live` + `/health/ready` + `/version` on `TEMPORAL_WORKER_METRICS_LISTEN` (default `127.0.0.1:9094`); `METRICS_ENABLED=true` also exposes `/metrics` |

**Not in this table:** Loki-only log pipelines are optional (`deployments/docker` observability profile). **ClickHouse** is optional cold-path analytics only (`cmd/worker` when `ANALYTICS_CLICKHOUSE_ENABLED=true`); Postgres remains authoritative—see `internal/platform/clickhouse` and `ops/ANALYTICS_CLICKHOUSE.md`. **Temporal** is optional: default off; when enabled, scheduler call sites in `cmd/api`, `cmd/worker`, and `cmd/reconciler` can enqueue workflow follow-up, and `cmd/temporal-worker` executes the registered task queue. Compose `experimental` profile can still run a Temporal server for local use.

**Observability:** worker, reconciler, mqtt-ingest, and temporal-worker keep health/readiness/version on their loopback ops listener even when metrics are disabled. When `METRICS_ENABLED=true`, bind that same listener to an address reachable from your scraper (e.g. `0.0.0.0:9091` when Prometheus runs in Docker; see `deployments/docker/README.md`). `cmd/api` keeps the legacy public health endpoints and can additionally mirror the same ops surface on `HTTP_OPS_ADDR`.

**2-VPS deployment note:** `cmd/reconciler` and `cmd/mqtt-ingest` should still be treated as singleton-like unless you intentionally add broker/shared-subscription or coordination semantics above the current defaults. `cmd/worker` is safer to scale horizontally because the outbox and JetStream paths are already designed around database/queue semantics, though duplicate work attempts are still possible and must remain idempotent.
